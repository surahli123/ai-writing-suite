# Code Review Skill

Conduct a thorough code review for quality, security, and maintainability with severity-rated feedback.

## When to Use

This skill activates when:
- User requests "review this code", "code review"
- Before merging a pull request
- After implementing a major feature
- User wants quality assessment

## GPT-5.5 Guidance Alignment

- Default to outcome-first progress and completion reporting: state the target result, evidence, validation status, and stop condition before adding process detail.
- Treat newer user task updates as local overrides for the active workflow branch while preserving earlier non-conflicting constraints.
- If correctness depends on additional inspection, retrieval, execution, or verification, keep using the relevant tools until the review is grounded; stop once enough evidence exists.
- Continue through clear, low-risk, reversible next steps automatically; ask only when the next step is materially branching, destructive, credentialed, external-production, or preference-dependent.

Delegates to the `code-reviewer` and `architect` agents in parallel for a two-lane review:

1. **Identify Changes**
   - Run `git diff` to find changed files
   - Determine scope of review (specific files or entire PR)

2. **Launch Parallel Review Lanes**
   - **`code-reviewer` lane** - owns spec compliance, security, code quality, performance, and maintainability findings
   - **`architect` lane** - owns the devil's-advocate / design-tradeoff perspective
   - Both lanes run in parallel and produce distinct outputs before final synthesis
   - If either lane cannot be launched or does not return evidence, report `independent review unavailable`; do **not** substitute the current/authoring lane, and do **not** approve or mark the review merge-ready.

3. **Review Categories**
   - **Security** - Hardcoded secrets, injection risks, XSS, CSRF
   - **Code Quality** - Function size, complexity, nesting depth
   - **Performance** - Algorithm efficiency, N+1 queries, caching
   - **Best Practices** - Naming, documentation, error handling
   - **Maintainability** - Duplication, coupling, testability

4. **Severity Rating**
   - **CRITICAL** - Security vulnerability (must fix before merge)
   - **HIGH** - Bug or major code smell (should fix before merge)
   - **MEDIUM** - Minor issue (fix when possible)
   - **LOW** - Style/suggestion (consider fixing)

5. **Architectural Status Contract**
   - **CLEAR** - No unresolved architectural blocker was found
   - **WATCH** - Non-blocking design/tradeoff concern that must appear in the final synthesis
   - **BLOCK** - Unresolved design concern that prevents a merge-ready verdict

6. **Specific Recommendations**
   - File:line locations for each issue
   - Concrete fix suggestions
   - Code examples where applicable

7. **Final Synthesis**
   - Combine the `code-reviewer` recommendation and the architect status into one final verdict
   - Approval requires explicit evidence from both independent lanes; missing or failed delegation is a blocking unavailable-review state, not an approval fallback
   - Deterministic merge gating rules:
     - If architect status is **BLOCK**, final recommendation is **REQUEST CHANGES**
     - Else if `code-reviewer` recommendation is **REQUEST CHANGES**, final recommendation is **REQUEST CHANGES**
     - Else if architect status is **WATCH**, final recommendation is **COMMENT**
     - Else final recommendation follows the `code-reviewer` lane
   - The final report must make architect blockers impossible to miss

     2	# sync.sh — Single-source → target assembler for AI Writing Suite packaging.
     3	#
     4	# SOURCE OF TRUTH: skills/ai-writing-suite/ (the directory above packaging/)
     5	# GENERATED:      packaging/claude/   packaging/codex/
     6	#
     7	# Rule: NEVER hand-edit files inside packaging/claude/ or packaging/codex/.
     8	#       Edit the source under skills/ai-writing-suite/ and re-run this script.
     9	#       See packaging/README.md for the full rationale (design R4).
    10	#
    11	# Usage:
    12	#   bash packaging/sync.sh            # from repo root OR from packaging/
    13	#   bash packaging/sync.sh --dry-run  # print what would happen, copy nothing
    14	#   bash packaging/sync.sh --check    # verify committed Codex body matches source; no writes (CI gate)
    15	#
    16	# Idempotent: safe to run multiple times; always produces the same result.
    17	# No external deps: pure bash + cp + mkdir.
    18	
    19	set -euo pipefail
    20	
    21	# ── Resolve paths ─────────────────────────────────────────────────────────────
    22	# Works whether invoked from repo root, from packaging/, or via symlink.
    23	SCRIPT_DIR="$(cd "$(dirname "${BASH_SOURCE[0]}")" && pwd)"
    24	PACKAGING="$SCRIPT_DIR"                        # packaging/
    25	SOURCE="$SCRIPT_DIR/.."                        # skills/ai-writing-suite/
    26	
    27	# Generated content destinations differ per target:
    28	#   Claude → packaging/claude/                       (content at marketplace root)
    29	#   Codex  → packaging/codex/plugins/ai-writing-suite/ (content inside the plugin;
    30	#            codex/ itself is the MARKETPLACE root, not the plugin)
    31	CLAUDE_TARGET="$PACKAGING/claude"
    32	CODEX_TARGET="$PACKAGING/codex/plugins/ai-writing-suite"
    33	
    34	DRY_RUN=false
    35	CHECK=false
    36	case "${1:-}" in
    37	  --dry-run) DRY_RUN=true; echo "[sync] DRY RUN — no files will be written" ;;
    38	  --check)   CHECK=true ;;
    39	  "")        ;;
    40	  *) echo "[sync] unknown arg: $1 (use --dry-run or --check)"; exit 2 ;;
    41	esac
    42	
    43	# ── Helpers ───────────────────────────────────────────────────────────────────
    44	do_copy() {
    45	  local src="$1" dest="$2"
    46	  if $DRY_RUN; then
    47	    echo "  copy  $src  →  $dest"
    48	  else
    49	    mkdir -p "$(dirname "$dest")"
    50	    cp -r "$src" "$dest"
    51	  fi
    52	}
    53	
    54	do_copy_file() {
    55	  local src="$1" dest="$2"
    56	  if $DRY_RUN; then
    57	    echo "  file  $src  →  $dest"
    58	  else
    59	    mkdir -p "$(dirname "$dest")"
    60	    cp "$src" "$dest"
    61	  fi
    62	}
    63	
    64	log() { echo "[sync] $*"; }
    65	
    66	# ── Read version from router SKILL.md frontmatter ─────────────────────────────
    67	# Frontmatter format: --- \n name: ... \n version: X.Y.Z \n ---
    68	# (version field is optional in v1; we default to 1.0.0 if absent)
    69	SKILL_VERSION="$(
    70	  sed -n '/^---[[:space:]]*$/,/^---[[:space:]]*$/ s/^version:[[:space:]]*//p' \
    71	    "$SOURCE/SKILL.md" | head -n1 | tr -d '\r'
    72	)"
    73	if [ -z "$SKILL_VERSION" ]; then
    74	  SKILL_VERSION="1.0.0"
    75	  log "no version: field in SKILL.md frontmatter — defaulting to $SKILL_VERSION"
    76	fi
    77	log "source version: $SKILL_VERSION"
    78	
    79	# ── What we copy ──────────────────────────────────────────────────────────────
    80	# Both targets receive the same logical content; only the plugin manifest
    81	# directory name differs (.claude-plugin vs .codex-plugin).
    82	#
    83	# Source layout (relative to $SOURCE):
    84	#   SKILL.md                    router
    85	#   skills/comms-polish/        sub-skill
    86	#   skills/comms-draft/         sub-skill (stub)
    87	#   skills/comms-qa/            sub-skill (stub)
    88	#   skills/voice-onboard/       sub-skill
    89	#   _shared/                    patterns, knowledge, voice-profile, etc.
    90	#   NOTICE.md  LICENSE  README.md  CHANGELOG.md
    91	
    92	SKILL_FILES=(SKILL.md NOTICE.md LICENSE README.md CHANGELOG.md)
    93	SKILL_DIRS=(skills _shared)

    95	sync_target() {
    96	  local target="$1"
    97	  local label="$2"
    98	  log "syncing → $label ($target)"
    99	
   100	  # Remove previously-generated content so deletions in source propagate.
   101	  # We only wipe the content dirs, never the manifest dir (.claude-plugin /
   102	  # .codex-plugin) because those contain hand-maintained manifests.
   103	  for d in "${SKILL_DIRS[@]}"; do
   104	    if [ -d "$target/$d" ]; then
   105	      if $DRY_RUN; then
   106	        echo "  rm -rf $target/$d"
   107	      else
   108	        rm -rf "${target:?}/$d"
   109	      fi
   110	    fi
   111	  done
   112	
   113	  # Top-level files
   114	  for f in "${SKILL_FILES[@]}"; do
   115	    [ -f "$SOURCE/$f" ] && do_copy_file "$SOURCE/$f" "$target/$f"
   116	  done
   117	
   118	  # Directories (recursive)
   119	  for d in "${SKILL_DIRS[@]}"; do
   120	    [ -d "$SOURCE/$d" ] && do_copy "$SOURCE/$d" "$target/$d"
   121	  done
   122	
   123	  log "  done → $label"
   124	}
   125	
   126	# ── Check mode: verify committed Codex body matches source (no mutation) ──────
   127	# Codex is the one host whose plugin BODY is committed (it resolves the plugin from
   128	# a physical path, so the body must be on the remote). This guards that committed
   129	# copy against drifting from the source tree. Non-mutating: builds into a temp dir
   130	# and diffs, excluding the hand-maintained .codex-plugin manifest.
   131	if $CHECK; then
   132	  log "check mode: comparing committed Codex body against source (no files written)"
   133	  TMP="$(mktemp -d)"
   134	  trap 'rm -rf "$TMP"' EXIT
   135	  for f in "${SKILL_FILES[@]}"; do
   136	    [ -f "$SOURCE/$f" ] && cp "$SOURCE/$f" "$TMP/"
   137	  done
   138	  for d in "${SKILL_DIRS[@]}"; do
   139	    [ -d "$SOURCE/$d" ] && cp -r "$SOURCE/$d" "$TMP/$d"
   140	  done
   141	  if diff -r --exclude=.codex-plugin "$TMP" "$CODEX_TARGET" >/dev/null 2>&1; then
   142	    log "check OK: committed Codex body is in sync with source."
   143	    exit 0
   144	  else
   145	    log "DRIFT: committed Codex body is STALE vs source. Run 'bash packaging/sync.sh' and commit."
   146	    diff -r --exclude=.codex-plugin "$TMP" "$CODEX_TARGET" | sed 's/^/  /' || true
   147	    exit 1
   148	  fi
   149	fi
   150	
   151	# ── Sync both targets ─────────────────────────────────────────────────────────
   152	sync_target "$CLAUDE_TARGET" "claude"
   153	sync_target "$CODEX_TARGET"  "codex"
   154	
   155	# ── Version lockstep check ────────────────────────────────────────────────────
   156	# Warn (not fail) if plugin manifests carry a different version than the source.
   157	# Update the manifests manually to match after bumping the source version.
   158	check_manifest_version() {
   159	  local manifest="$1" label="$2"
   160	  if [ ! -f "$manifest" ]; then
   161	    log "  WARN: manifest not found: $manifest"
   162	    return
   163	  fi
   164	  # Extract "version" field with python3 (available on macOS + Linux)
   165	  local mver
   166	  mver="$(python3 -c "
   167	import json, sys
   168	data = json.load(open(sys.argv[1]))
   169	print(data.get('version', ''))
   170	" "$manifest" 2>/dev/null || true)"
   171	  if [ -z "$mver" ]; then
   172	    log "  WARN ($label): could not read version from $manifest"
   173	  elif [ "$mver" != "$SKILL_VERSION" ]; then
   174	    log "  WARN ($label): manifest version ($mver) ≠ source version ($SKILL_VERSION)"
   175	    log "         Update \"version\" in $manifest to match."
   176	  else
   177	    log "  OK ($label): manifest version matches source ($SKILL_VERSION)"
   178	  fi
   179	}
   180	
   181	if ! $DRY_RUN; then
   182	  check_manifest_version "$CLAUDE_TARGET/.claude-plugin/plugin.json" "claude"
   183	  check_manifest_version "$CODEX_TARGET/.codex-plugin/plugin.json"   "codex"
   184	  # Note: $CODEX_TARGET now points at packaging/codex/plugins/ai-writing-suite/,
   185	  # so this resolves to the plugin manifest in its new conformant location.
   186	fi
   187	
   188	log "sync complete."

+# AI Writing Suite
+
+Polish prose that sounds AI-written without changing what the author meant.
+
+The differentiator is not "rewrite this better." It is:
+
+```text
+preserve meaning, remove AI slop, keep author voice
+```
+
+Use it when a draft has the usual model tells: filler, vague claims, inflated importance, forced structure, and rhythm that sounds too even.
+
+## What It Does
+
+`ai-writing-suite` is a four-part writing-assistant skillset:
+
+1. **comms-polish** (v1) — identify tells, preserve meaning, remove AI-shaped prose patterns, match author voice or register.
+2. **voice-onboard** (v1) — interview, collect writing samples, distill your voice signature.
+3. **comms-qa** (v2) — knowledge QA over a pluggable writing playbook.
+4. **comms-draft** (v2) — guided drafting workflow using playbook best practices.
+
+An agent using the suite can:
+
+1. Identify the audience and purpose
+2. Preserve facts, citations, numbers, commands, and claims
+3. Remove AI-shaped prose patterns
+4. Match the author's voice or the document's register
+5. Return either a rewrite, a review, a detection score, or targeted file edits
+
+## Engine vs Fuel
+
+The OSS version ships the **engine** (comms-polish, voice-onboard, eval harness, self-improvement) with a generic **knowledge base** distilled from four open-source projects.
+
+The company version reuses the same engine and swaps a proprietary **DS Comms Playbook** into the knowledge slot via a Confluence page. The playbook never enters this public repository.
+
+```
+OSS:     engine + generic KB
+Company: engine + proprietary playbook (dropped in as data, not code)
+```
+
+No code changes needed to switch playbooks.
+
+## When To Use It
+
+Use the suite for:
+
+- docs and README prose
+- emails and status updates
+- social posts and personal notes
+- reports and summaries
+- launch notes and user-facing copy
+
+Do not use it for source-code cleanup. Use a code cleanup or refactoring skill for that.

+## Sub-Skills
+
+### comms-polish (Ships in v1)
+
+Polish prose and detect AI tells.
+
+**Modes:**
+- `detect` — find AI tells and estimate density without rewriting.
+- `rewrite` — produce polished prose from pasted text.
+- `edit` — modify a prose file in place while preserving structure.
+- `review` — give prioritized writing findings without rewriting everything.

+**Features:**
+- Consolidated AI-tell pattern catalog (seven deduplicated sources, each pattern source-tagged for attribution).

+Interview and distill your writing voice.

+Knowledge QA over a pluggable playbook.
